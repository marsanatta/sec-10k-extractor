## Part 1 — Key principles from the workshop

### P1. A single pass/fail score hides everything that matters
Two agents can both "pass" a task while being completely different stories: one took fewer steps, cost less, crashed zero times; the other thrashed (Balaji, ~11:50). Arvind's whole talk is that **capability ≠ usefulness**: over 18 months accuracy on benchmarks rose steeply while **reliability rose only gradually** — "yes, capability is going up, but capability doesn't fully measure usefulness" (~27:00–29:00). Takeaway: report *multi-dimensional* metrics, not one headline number. Mahesh's version: success rate is the *start*, then layer efficiency (steps), token count, latency, **cost**, and safety (~1:08).

### P2. Reliability has nameable sub-dimensions — measure each
Arvind decomposes reliability into **consistency** (outcome consistency: same pass/fail each run; trajectory consistency: same actions each run; resource/cost consistency), **robustness** (fault robustness — inject API timeouts/errors; prompt robustness — LLM-reword the prompt preserving semantics), **calibration** (elicited confidence should match empirical success rate), and **severity of failure** (formatting error vs data deletion) (~26:40–32:10). Key nuance: companies fixed *overconfidence* (sycophancy backlash) but **discrimination got worse** — agents now say "~50%" for everything; you want confidence that *separates* successes from failures, not just well-calibrated-on-average (~30:00–32:00).

### P3. "70% accuracy" is ambiguous until you ask *which 70%*
Either the agent reliably nails 70% of tasks and reliably fails the other 30% (predictable, deployable), or on *any* task it works 70% of the time stochastically (un-deployable for automation) (~26:48). This is the consistency distinction and it changes the deploy decision.

### P4. Automation vs augmentation changes your reliability bar
"If Alexa plays the wrong song 10% of the time that's an annoyance; if an agent uses your credit card and orders wrong 10% of the time, that's dead on arrival" (~24:55). Coding agents tolerate errors because a human reviews the diff (augmentation); autonomous customer service does not (automation) (~33:40–34:20). Release decisions should gate on a **reliability threshold**, not capability alone.

### P5. Eval-first, not eval-last — and don't start at the wrong altitude
Mahesh's core advocacy: enterprises lack an "eval-first mentality" — they ship, then look only at the final output, "and quickly you run into issues where you change something and you don't understand what failed" (~57:00). Two named traps: (a) **starting too granular** — auditing "is the planner's function call right?" before you've measured end outcomes; that comes *eventually*, not first (~57:40); (b) **"deploy to production and evaluate"** (~58:46). Borrow from software: unit tests, regression tests (~58:00).

### P6. Climb the verifiability ladder; use LLM-judge only where you must
- **Level 0 — verifiable** (Mahesh ~59:54): coding "did the unit test pass?", math "is the value correct?". Prefer this. **Gotchas even here**: not all unit tests deserve equal weight (a few important tests vs many trivial ones — agent passes most yet does no real work); and **reward hacking** — "the agent fixes the unit test so it artificially passes instead of fixing the bug" (~1:00:30–1:01:17).
- **Level 1 — non-verifiable** (deep-research paragraph output, no single right answer): **rubrics** decomposed into 0/1 criteria, each scored by **LLM-as-judge**, then weighted into a score (~1:01:21–1:03:00).
- **Hard vs soft verifiers** (Pierre, GAIA 2, ~1:21–1:23): GAIA 2 deliberately **moved away from LLM rubric-judging toward event verification**. A **hard verifier** checks tool-call parameters with "pure equality or simple algorithmic code" — "a lot cheaper than calling an LLM and a lot more reliable to reproduce." A **soft verifier** (LLM) is used *only* where content needn't match exactly, e.g. "is the email body roughly the one we expected." Principle: **program the check whenever the correct answer is mechanically decidable; reserve the LLM for genuinely fuzzy content.**

### P7. Verify against an *expected diagram of actions*, not the final blob
GAIA 2 annotates each scenario with the expected set/order of actions, then compares the agent's actual action graph event-by-event: right actions, right order, right parameters (~1:50–1:53). This is structural ground truth derived from the task spec, not a learned label.

### P8. Build the eval so it can't be gamed / overfit
- **Reword tool signatures and inject tool failures** while keeping the task fixed, "so the agent is not overfitting over specific signatures" and can't "guess what the tools look like" (Pierre, ~48:50, ~1:42). Robustness testing *is* anti-overfitting.
- **Strict checkers** that explicitly look for "hacking patterns" (Pierre, ~1:36).
- **The agent must not have access to the grader/solution** — Harbor enforces this natively, "otherwise it can reward hack" (Mahesh, ~1:06:17). And **grade everything you asked for**: "if you ask for three things but grade only two, the agent learns it doesn't need to do the third" (~1:10:00). Add a fingerprint to unit tests so the agent can't muck with them (~1:11:20).
- **Saturation ≠ publicity.** Balaji: the common belief that public benchmarks saturate faster is *not* empirically supported (his lab's saturation paper finds public/private/partially-private has no strong correlation with saturation speed, ~1:38:57); meanwhile **secrecy blocks verification**, so he favors disclosure + "the bare minimum information for someone else to validate your eval" (~1:39:50).
- **Saturation may be the *metric*, not the task.** "What if what's getting saturated is not the task suite — the tasks are fine, but the *metric* is saturated?" You exhaust a 1-D capability score long before you exhaust the 12 dimensions of reliability/cost/latency (Arvind, ~37:00).

### P9. Reproducibility demands session-level + full system disclosure
"Agent identity is a black hole — most agentic evals are tagged with a model name. The model is not the agent" (Balaji, ~14:30). Report sub-agent list, MCP servers, memory config, scaffold/harness, seeds, prompt perturbations, **pass@k**, and **cost** — "cost reporting is inconsistent or absent" (~14:30–15:30). Same model on different harnesses gives drastically different success rates (Mahesh, terminal-bench, ~1:10:14). Nathan's corollary: **don't evaluate through an inference provider** — "in reality you are evaluating the provider," you don't know how it's prompted in the back end, it's not reproducible; use local/controlled inference (~1:27:00).

### P10. Error analysis loop: read traces, then automate trace analysis
"Read your traces" to feel how the agent behaves, then run **automated failure analysis on the trajectories** to surface where success rate drops — "the agent is generally good, but in *this specific kind of scenario* the success rate drops" (Mahesh, ~1:11:37–1:12:30). Arvind's manual+automatic trace analysis on GAIA surfaced concrete failure modes: models conflate a **messy process** (tool-call failures) with a **wrong answer** even when the answer was right (a calibration bug); and **ambiguous questions** — bad for eliciting capability, but *gold for reliability*, because real deployments are full of ambiguity (~32:30–33:20). Faults (making needed data inaccessible) **induce hallucinations** that don't appear by default (~33:40).

### P11. Isolated, reproducible, *cheap* environments — and the sim-to-real gap
Build a sandbox that holds dependencies + world-state + tools, so eval is repeatable, **safe** (you can test destructive actions — "delete all my emails" — without touching the real world), and **cheap** (don't depend on external bandwidth/APIs, so it's reproducible) (Mahesh ~1:04, Pierre ~44:50). The standing tension is the **sim-to-real gap**: the sandbox must mimic production closely or the numbers don't transfer (~1:14:50). GAIA 2 also **simulates time** — fast-forward weeks so a "wait for the price to drop" task doesn't take weeks (~47:50).

### P12. Living benchmarks: evolve the set as the system improves
"Numbers change very quickly with new models" — GAIA 2 already had to **add turns and expectations** and **re-tune the judges**, because "the output style we expected from models last year is very different this year" (Pierre, ~52:24). Nathan's Community Evals makes benchmarks *living*: an `eval.yaml` per dataset, results submitted as PRs, leaderboard auto-updates, the community can dispute a score on the PR, and people fork/extend the eval set the way they fork models (~1:22–1:48). Reuse an existing eval framework (inspect-AI, LightEval) rather than rolling your own — "way harder to maintain your own" (~1:24:46). Always log standard error of the eval (~1:29:00).

### Net new actions this workshop surfaced (not yet in 03/06/08/10)
1. **Run-to-run consistency + prompt-perturbation robustness** for the non-deterministic LLM-LIB layer (P2/P8) — measure boundary-stability and flag-stability across k runs and reworded prompts. Currently we test format variance across *filings* but not stochasticity across *runs of the same filing*.
2. **Cost + latency reported alongside accuracy** in the standard eval table (P1) — pull the `05` numbers into `08` §D.
3. **A `run_manifest`** (model build, temp, seed, few-shot hash, code commit, harness version) emitted per eval run (P9) — reproducibility is currently assumed, not enforced.
4. **Automated failure-cluster table** grouping low-confidence filings by (era × pathology × signal-fired) (P10), and an audit that the confidence aggregator doesn't penalize correct boundaries for noisy-but-irrelevant heuristic failures (Arvind's messy-process bias).
5. **Style/whitespace perturbation of gold filings** as an explicit overfit test (P8), mirroring GAIA 2's "reword tool signatures."
6. **Judge re-calibration trigger** on every model-build change (P12), since "looks like Item 7" drifts with the judge model's version.

- **Hard programmatic checks > LLM-judge** wherever decidable; LLM-judge as one fuzzy vote only (P6) — exactly `03`/`06`.
- **Independent oracles, never self-consistency** (P6-hack) — exactly our CLAUDE.md rule and `06`/`08`.
- **Expected-item template from DEI/date** as structural ground truth (P7) — exactly `06` step 4.

## One-line synthesis
The workshop's spine — *don't trust a single score, prefer hard programmatic verifiers, reserve the LLM-judge for genuinely fuzzy content, validate against independent oracles, make the eval un-gameable, measure run-to-run reliability and cost, read the traces, and evolve the set* — is almost exactly our existing plan, with three genuine additions we were missing: **run-to-run/prompt robustness, an enforced run-manifest for reproducibility, and cost reported next to accuracy.**
